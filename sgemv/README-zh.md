# SGEMV kernel
矩阵-向量乘法，常见用法有LLM 模型推理中的Decode阶段，上一步生成的token在不同步骤中与weight及kv cache做gemv计算
以`N * N`的矩阵和`N * 1`向量的乘举例，计算结果为`N * 1`的向量，其每个元素计算需要进行`N`次乘法，`N-1`次加法，一共要算`N`个元素，总的计算量是`N * (N + N - 1)`，约等于`2N^2`
计算过程中，需要读矩阵`N * N`和向量`N * 1`，需要写向量`N * 1`，若每个元素以 4Bytes 的标准`float`存储，总访存便是 `(N * N + 2 * N) * 4`
这时的计算访存比为 `2 * N^2 / 4 * (N^2 + 2 * N) = N / 2 * (N + 2)`，矩阵规模越大，结果越趋近于 `1/2`，是访存密集型算子

## ¿ How to optimize ¿
不同于计算密集型算子的优化思路，访存密集型算子的瓶颈不在并行度上，优化这类任务的主要思路是如何用满带宽、读取的数据如何多次使用，或是如何减少要读的数据量(废话，哪个算子不需要这些优化)
因此，我们的优化手段也比较明确了，能用向量化访存就尽可能用，尽可能的合并访存，尽可能的利用硬件资源
因为把数据从global mem搬到shared mem，再通过shared mem到register会增加一次搬运，且对shared mem的访问可能有bank conflict，使用shared mem能不能带来性能提升需要测试
GEMV的计算中，M * N矩阵的每个元素都只用一次，对其用shared memory没什么意义，N向量却一直在用，预取一下它，看看对比直接读global mem能不能带来提升
